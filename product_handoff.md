## 一、产品定位

### 核心问题
年轻人刷到让自己焦虑的信息后，陷入"想做点什么但什么都做不了"的瘫痪感。根本原因不是压力太大或计划不够，而是**和自己的生活失去了真实连接**。

### 一句话定位
> 当你被外界煽动焦虑时，打开它，跟着呼吸，平复情绪后做一件关照自己身体或丰容生活的小事，重新掌控自己的生活。

### 差异化立场
市场上大多数产品（冥想类、习惯养成类）的隐含逻辑是"你要不断变好，否则落后"——在制造焦虑。
**这个产品的立场相反：外部世界的变化与你无关，你对身体和生活的感受永远是你的。**

- 竞品是在焦虑赛道上卷；这个产品说这条赛道本身有问题。
- 冥想类产品让你平静下来；这个产品让你平静之后有地方去。
- 别人解决焦虑发生之后的问题；这个产品解决**焦虑发生那一刻**的问题。

### 面试口径（作品集用）
> "我在解决的是一个很具体的状态：年轻人刷到让自己焦虑的信息之后，陷入一种'想做点什么但什么都做不了'的瘫痪感。市面上的产品要么帮你冥想放松，要么帮你规划目标，但这个状态的根本原因不是压力太大也不是计划不够，而是和自己的生活失去了真实连接。所以我做的事是：当你被外界煽动的时候，给你一件此刻就能做的、和自己身体或感知有关的小事，帮你重新落回自己的生活里。"

## 二、目标用户

- **年龄**：16–26岁，在校学生或职场新人
- **核心痛点**：缺乏内在坐标系，容易被外部信息触发方向焦虑
- **核心画像**："躺又躺不平干又干不动，感觉有一万件事等着我但提不起劲规划。"
- **使用动机**：不是主动想提升自己，而是需要在混乱中找到一个落脚点，或为无聊的生活带来意义感。

---

## 三、理论基础

### 丰容（Enrichment）概念
借自动物学：动物园里的动物因环境单调、刺激缺失，会出现刻板行为（踱步、咬笼子）。
人类的"刻板行为"（无休止刷手机、熬夜、摆烂）同理——不是意志力问题，是**环境贫乏问题**。
丰容把"我怎么又堕落了"这个自责问题，转化为"我的生存环境给我提供了什么刺激"这个设计问题。

**动物丰容五大维度（迁移到人类）：**
- 食物丰容
- 环境丰容
- 感知丰容（视觉、听觉、嗅觉、触觉）
- 认知丰容
- 社群丰容

## 四、真正丰容的判断原则（不变）

### 原则一：感知是否真实发生
必须有什么真的通过感官/身体/情绪抵达你。
信息增加 ≠ 丰容，感知拓宽才是。
**检验**：做这件事的过程中，有没有什么真的抵达你？
> 注：感知是否发生以用户体验为准，不限于身体感官——看纪录片感受到自然和人文之美，也是真实的感知。
> 注2（v3 补充）：同一件事重复做也可能产生新的真实感知（光线、心情、状态每次不同），陌生感不等于"内容本身没见过"，这是推送层"池子耗尽后允许重置"决策的理论依据。

### 原则二：轻微陌生感，无表演压力
陌生感是激活条件；但如果主要在想"我做得好不好"，就滑向表演，本质还是在回应外部评价体系。
**检验**：做这件事时，你在意的是过程的感受，还是结果的呈现？

### 关于打卡出片（重要澄清）
打卡出片不是丰容的敌人。
- ✅ 感知先发生，拍照是延伸 → 丰容
- ⚠️ 感知和表演动机同时存在 → 体验稀释，但不为零
- ❌ 拍照替代感知，出片是目的，体验只是道具 → 伪丰容

### 关于"完美主义"的产品态度（重要）
画速写、找"最好角落"这类活动，必须在产品文案中传递：**做不完美也没关系，重点是体验过程。**
人类不是AI，这个过程不必有结果，也不必对谁负责。
产品的初衷是让用户更多体验自己、自己的生活，温婉地包容自己。
这个语气应渗透进每一个交互细节，不只是一句文案。

## 五、内容分层架构

### 5.1 两个完全独立的内容层

```
┌─────────────────────────────────────────────────────────┐
│  推送层（通用、扁平、不分类）                              │
│                                                           │
│  所有条目共同特征：时间短 + 不挑场景/工具 + 随时能做         │
│  → 不属于任何图鉴，不需要解锁，新用户第一天就能用全部         │
│  → 只按 scene_tags 过滤（11 维具体场景，见 content_principles.md §五），与图鉴解锁状态无关  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  丰荣图鉴层（需要更多投入，分门别类，需要解锁）              │
│                                                           │
│  每个图鉴内是更长、更需要条件的内容（原"探索池"内容）         │
│  用户主动去"丰荣探索"tab 逛、挑、做，自己攒解锁名额           │
└─────────────────────────────────────────────────────────┘
```

**关键收益**：
- 日常主线体验（开机仪式）完全不依赖图鉴解锁状态，新用户冷启动问题自动解决
- 不再需要决定"该从哪个图鉴里推送"——推送压根不从图鉴抽

**权衡（已知，暂不处理）**：日常推送和图鉴收集彼此独立，做了很多次推送不会让图鉴有任何进度。决定保持这种独立，不做桥接提示。

### 5.1.1 推送层去重、刷新与池子重置机制（v3 新增）

**为什么需要去重**：原则二要求"轻微陌生感"才能激活真实感知，如果同一条内容连续/频繁推给同一用户，陌生感会被磨掉，体验滑向机械执行任务。

- **去重规则**：排除该场景下所有"已做过"（已点击"做完啦"）的内容，无论该场景内容池总量多大，都严格排除，直到该场景下全部内容都做过一轮
- **"换一个"刷新功能**：用户对当前推送的内容不感兴趣时，可点击刷新，从该场景"未做过"的池子中重新随机抽取一条；刷新本身不会把内容标记为"已做过"，所以同一条可能被多次刷新到（如刷新出 B、D、C，最终又刷回 A），刷新顺序需保证随机
- **池子耗尽后的处理**：当某场景下所有内容都做过一轮后，**直接重置**（清空该场景下"已做过"的排除记录，重新开放全部内容随机抽取），**不做任何提示**，用户无感知。理由见原则一注2：重复做同一件事仍可能产生真实的新感知，不需要为此设计额外仪式或提示（数值不可见原则同样适用于这层隐藏状态）
- **刷新次数限制（v4 新增）**：每次推送的"换一个"刷新最多3次。达到上限后不再继续刷新，前端展示一段语气文案，大意是反复刷新可能意味着此刻状态疲惫/有压力，不一定要做点什么，可以放下手机、闭眼休息一会——把"限制"重新框成一句关心，而不是系统性拒绝。这条限制也是为了不让"换一个"被滥用成一个隐性的内容选择器，违反"推送而非选择"的原则

### 5.2 图鉴（原"主题月"，已改名）

- **改名理由**：避免"月"带来的时间紧迫感；改为类似图鉴/收藏的形态，无 deadline，按完成度推进
- **形态**：用户主动选择，知道自己在收集什么（保留 v1 的自主感原则）
- **完成奖励**：完成度达到 100% 时触发"回顾"（见 5.4），**回顾本身即奖励**，不设计额外勋章/徽章

### 5.3 图鉴激活机制（v7 重设计：移除名额体系）

图鉴墙默认全部锁定。用户点击任意图鉴卡片，展开详情（图鉴名 + 简介 + 一条示例内容的 title + hook），决定是否开始探索。

**激活流程**：

```
用户点击锁定图鉴
        │
        ▼
底部展开：图鉴名 + 简介 + 示例条目（title + hook）
        │
        ├── [开始探索] ── 当前激活数 < 3 → 直接激活，进入图鉴内容页
        │                当前激活数 = 3 → 展示上限提示（见下）
        └── [再逛逛]  ── 收起，继续浏览图鉴墙
```

**上限提示文案**（第4次点开时，在展开卡片内显示）：
> 过多的选择可能分散对过程感受的注意力。你随时可以开启新的图鉴，不过需要先把一个暂时放下。

下方列出当前3个激活中的图鉴，每个旁边有 **[放下]** 按钮，用户选择放下一个后 [开始探索] 变为可用。

**"放下"规则（v7 新增）**：
- 任何激活中的图鉴（无论是否已有进度）均可放下，无惩罚
- 放下后：图鉴回到锁定外观，已完成条目的绿色状态保留在内部；再次进入时仍可见
- 放下不消耗任何资源，随时可重新激活（需再占一个激活位）

**完成后自动释放**：图鉴完成度达到100%并触发回顾后，自动不再占用激活位，用户无需操作。

**未激活图鉴的呈现**：图鉴墙上显示名称和简介预览（类似图鉴轮廓），不显示具体条目内容。

### 5.3.0 同时激活图鉴数量上限（v7 更新）

**上限为 3 个**（v4 方案为 4 个且依赖名额体系；v7 简化为 3 个固定上限，名额概念已移除）。

设定 3 的理由：让用户在有限的探索空间里保持专注，避免"同时打开多本但都浅尝辄止"——这和"真实感知需要投入"的丰容核心一致。3 也是新用户首次可激活的数量，保持逻辑统一。

超出上限的处理见 5.3。

### 5.3.1 图鉴完成度计算方式（v3 确认）

完成度 = **该图鉴下"做过的不同条目数" / "图鉴总条目数"**。重做同一条目**不**增加完成度。

**为什么这样算**：完成度这个隐藏数值唯一的作用是决定"什么时候触发回顾"，本质回答的是"用户在这个图鉴里投入够不够深"。如果重做也能加分，会变相鼓励反复刷最简单的条目，这正是"丰容不是习惯养成"原则要避免的。按不同条目数计算，逼迫用户真正拓宽体验广度。

**内容库版本变更的影响（v4 新增，v7 更新）**：图鉴内容条目可能随运营持续增补，分母会变化。完成度阈值判断（100%触发回顾）采用**单向棘轮机制**：已生成的回顾不会因后续分母变大、完成度回落而被撤销；图鉴在丰荣探索里的完成度展示始终按当前真实的"做过的不同条目数 / 当前总条目数"计算。

### 5.4 回顾机制（替代原"月度回顾"）

- **触发条件**：某个图鉴完成度达到 100%
- **快照定格**：回顾基于触发那一刻的数据生成一次，不再"活的"更新
- **回顾后被动归档**：生成回顾后，该图鉴下当时仍未归档的对话，被动统一归档（因为快照已生成，没必要继续挂着）
- **鼓励多聊天**：为了给回顾积累更丰富素材，鼓励用户每完成一个丰容内容都去"聊聊"，但**安静、不主动催促**（不弹窗提醒、不做未聊天次数的红点/进度提示，只在入口文案上保留一点温度）

#### 5.4.1 回顾内容生成机制（v3 新增，核心设计）

**回顾 = 一段由AI生成的纯文本整体叙事，不附加任何图片或时间线/摘要列表。**

- **生成素材**：不读取原始对话记录，而是合成该图鉴下已生成的各次"完成摘要"（见 6.5.2），成本低（输入是若干条≤100字摘要，而非完整对话）
- **决策过程（重要，记录下来避免重复纠结）**：
  - 曾考虑在叙事下方附加"摘要+图片时间线"作为细节/对比展示，理由是"支持对比、复用已有数据、给叙事证据感"。后否决：这其实是把"图鉴网格里的完成次数计数UI"（5.7节，浏览图鉴时的展示）和"回顾该长什么样"搞混了；一段沉浸式叙事后面突然接一个数据列表，会打断"一起回望"的连续体验感，变成"抒情开场白+数据报表"的割裂感
  - 曾考虑把图片编织进叙事文字本身的引用里（"那次你拍的那张照片里…"），后否决：实现上需要生成调用具备视觉输入能力 + 模型输出携带结构化标记 + 前端解析渲染，复杂度上升一档，收益（叙事更生动）相对模糊，性价比不高
- 最终方案：**回顾就是叙事本身**，原始对话不设独立查看入口

**同一内容多次完成的叙事处理（v4 新增，呼应 5.7）**：图鉴下某个内容条目可能被反复重做多次，每次通过「说完了」退出（仅图鉴层）都会产生一条独立的完成摘要（见 6.5.2）。回顾生成时不机械列出或截断这些摘要，而是把"同一内容id下出现多条完成摘要"当成需要识别和归纳的一种模式，写入生成叙事的 system prompt 里要求 AI 主动呼应（例如"你完成过这件事x次，其中几次记录了某种相似的感受，你似乎对这件事有特殊的偏好"）。不对完成摘要的留存数量设上限——重做次数本身不需要单独的统计字段，可直接从已有的(内容id+完成时间)摘要记录中派生。

**不做结构化情绪统计/聚类（v4 新增，写入未来 System Prompt 设计的约束清单）**：模式识别不通过预先对摘要做情绪分类/打标签/聚类（如"7次平静、5次喜悦、2次失落"这类频次统计）实现——这本质上和已被否决的"叙事+数据列表"是同一种问题，会把"一起回望"的语气割裂成"心理评估报告"。正确路径是把全部原始摘要文本直接交给生成回顾的模型，由模型在写叙事的过程中自己读出重复出现的感受/语气并用语言表达（类似动力学咨询中"点出模式"的方式，是定性的、融合进叙事的，不是定量枚举）。这条约束本身不需要现在写出完整 prompt 文本，留待第九节第5条"System Prompt整体改写"阶段统一处理，但需要作为该阶段必须遵守的具体要求被记录下来，避免被忽略而误做成结构化情绪分类 pipeline。

#### 5.4.2 回顾的发现路径（v4 新增）

**发现路径**：回顾在图鉴完成度100%时**静默生成**，不打断当时"做完啦→可选聊聊"的轻量流程（完成最后一条目时，依然走正常的可选聊聊路径，不会被强行打断成全屏仪式）。生成后，在"丰荣探索"图鉴总览对应的图鉴卡片上加一个提示。用户点进提示才会看到回顾——如果点进去时还在生成中，前端展示邀请式的加载文案（如"一起回顾你为生活带来的新内容吧"），不用系统进度提示的语气。


- **语气边界（防止违反"不评价"人格原则）**：
  - ✅ 可以**反映/呼应**用户自己说过的内容，帮他看见连接（"你提到了xx、yy这些感受，这些探索好像确实给生活带来了一些不一样的触动"）
  - ❌ 不能**评价表现**（"你坚持完成了""你做得很好""你进步了"）
  - 必须传达"完美也罢、做不好也罢都没关系，这只是探索不是任务"的立场，但**这句话不是前端写死的固定模板**，而是写入生成这段叙事的 system prompt 里反复强调的立场要求，每次生成的具体措辞可以不同
  - 整体语气目标：类似心理咨询中"被陪伴、被看见"的感觉——不是总结报告，是一起回望
- **摘要素材为空时的兜底（v4 新增）**：如果该图鉴所有完成事件用户都跳过了"聊聊"，没有任何完成摘要可用，回顾叙事改用纯陪伴语气模板——不复述具体做过哪些条目，不推测过程或感受，只传达"这段经历真实存在、不必被说出来才算数"的态度（例如：「有一段时间，你在＜图鉴名＞这件事上，认真地走了一段路。具体发生了什么，我们没能听你讲起过，但这并不妨碍这段经历真实地属于你。有些感受不必说出来才算数，安静地经历过，本身就足够了。」）。这个分支宁可牺牲不同图鉴之间的区分度，也不让AI编造用户没说过的细节或感受

### 5.5 图鉴命名维度（混用，不强求统一逻辑）

不再用纯感官学术词（"视觉感知图鉴"太硬、没娱乐性），混用两条轴：

```
轴一：感官底层拆得更细更具体
  视觉 → 颜色图鉴 + 光影图鉴
  触觉 → 温度图鉴 + 质地图鉴
  听觉 → 城市声音图鉴 + 安静图鉴
  嗅觉味觉 → 气味图鉴 + 喝一口图鉴
  本体感觉 → 重量图鉴 + 平衡图鉴
  环境综合 → 角落图鉴 + 路线图鉴

轴二：以生活场景/对象为单位（不分感官，混合体验）
  手图鉴 / 阳台图鉴 / 通勤图鉴 / 厨房图鉴 / 天气图鉴 / 深夜图鉴
```

两条轴混用，避免图鉴墙看起来像"体检报告分类表"。

### 5.6 新用户首次进入丰荣探索（v7 重设计）

新用户第一次打开丰荣探索 tab，顶部显示一句说明文案（**仅此一次，用户点击任意图鉴卡片后消失**）：

> "这里是不同主题下的丰荣活动合集，挑感兴趣的自由探索。"

之后流程与所有用户完全一致（见 5.3）：浏览图鉴墙 → 点击感兴趣的卡片 → 看名称 + 简介 + 示例条目（title + hook）→ 决定是否开始。前3个图鉴直接激活，无任何前置条件。换选、放下规则与常规流程相同，不设独立的"首次解锁"机制。

### 5.7 重做同一丰容内容

- 图鉴内容可重复做，每次重做规则与首次相同
- **UI 保留"点亮/未点亮"二元状态，不展示重做次数**（v4 补充梳理，作废此前"升级为完成次数计数"的方向）：重做的价值不需要被证明或被看见——即使重做不算"陌生"、不增加完成度，体验本身（如反复去看同一片河面落日）依然有意义，这种价值不该被一个要求"被看见/被比较"的数字承载；数字角标天然容易被默认解读成"积累/成就感"（越大越好），与丰容不是习惯养成、数值不可见原则的精神冲突。"重逢感"完全交给对话与回顾叙事承载（见 6.5.2、5.4.1），不在UI层做任何数字/次数展示
- 重做时默认开新的独立对话窗口（见第六部分），但会传入上次同一内容的"完成摘要"（见 6.5.2），让 AI 有连续性意识
- 不对重做次数或完成摘要留存数量设上限——多次完成的模式识别交给回顾生成机制处理（见 5.4.1），不需要截断/采样工程

## 六、核心流程与对话机制

### 6.1 完整流程（v6 更新：同步当前实现形态）

```
打开产品
  ↓
呼吸引导（可跳过；屏幕上方常驻一句产品语境说明：
"这里每天有一些小事，去做做看，感受一下就够了。"）
  ↓
地球 online 卡片弹出
（每日首次打开自动触发；当天划走后可从主页常驻"今日任务候选"按钮随时重新打开）
（新用户首次进入：先填写基本信息 + scene_tags，保存后再展示卡片）
  ↓
卡片内容：城市+天气信息 + 今日任务候选列表
（按用户 scene_tags 过滤+去重随机抽取；可"换一批"最多3次；第3次用完后显示兜底关怀文案）
  ↓
领取一个任务 → 任务进入主页"已领取"列表
  ↓
去做
  ↓
回到App，在已领取列表点击"做完啦"（显式自报，无验证无评价，与是否聊天完全解耦）
  ↓
弹出可选邀请（前端固定文案，见 6.5.1）："要不要聊聊"
  ↓
（可选）进入独立对话窗口
  ↓
完成的任务当天显示在主页"今日已完成"区块（灰色，带"聊聊"按钮可直接进入对话）
次日，已完成任务在地球 online 卡片 greeting 后展示；用户可选择"聊聊"或"全部清掉"
  ↓
（独立路径）"丰荣探索" tab：浏览/解锁/做图鉴内容，完成后同样走"做完啦→可选聊聊"流程
```

### 6.2 导航结构

```
两个主 tab + 一个不算 tab 的角标：

  当下                      丰荣探索
（呼吸+场景+推送，           （图鉴全貌浏览、解锁、
  唯一主线动作，              做图鉴内容，
  原名"现在"）                原名"看看"）

                                        [角标：设置/账号]
```

- 不设常驻"回顾" tab、不设常驻"设置" tab——回顾按图鉴完成事件触发出现，设置类低频操作收进角标，减少主导航的信息密度
- 命名理由：避免"现在"过于口语化的紧迫感，"当下"更贴合呼吸/正念语境；"看看"换成"丰荣探索"以承载图鉴系统的身份感

#### 6.2.1 "设置"角标范围（v4 补充梳理新增）

通过一轮针对性 grill-me 梳理确定，"设置"不是另一个需要设计的体验入口，应保持低存在感：

- **主动提醒** ✅ 做，但按微信小程序「订阅消息」能力的真实限制设计，不是传统后台 push：
  - 不能随时主动推送，必须先有用户的一次主动触发（如"做完啦"之后）弹出授权询问，用户同意才能补发一条
  - 一次性订阅消息只保证能补发一次，无法保证长期重复提醒；长期订阅消息仅开放给政务/医疗/交通/金融/教育类公共服务类目，本产品类目不适用
  - 消息文案受微信模板库审核限制，不能完全自由发挥"温婉包容"语气，只能在模板允许范围内选最贴近的措辞
  - 订阅状态的开关由用户在微信「服务通知」里管理，小程序内"设置"页大概率只能放一个引导性的"去开启提醒"入口，做不到产品自己可控的开关
  - 已征得确认：接受这个落差，按订阅消息能力设计，不追求传统 push 体验
- **数据管理（清除/重置进度）** ❌ 不做：不留入口能让 5.3 节"已开始的图鉴不可放弃"规则没有后门，逻辑自洽；暴露"清空进度"功能本身暗示"进度是需要被管理的资产"，与"数值不可见、不强调完成率"的语气冲突。MVP 阶段开发者自测可直接清 localStorage，不需要做成产品功能。（留给生产阶段评估的潜在风险：若涉及账号注销/个人数据导出合规要求，应归入"账号"角标而非"设置"，性质不同，不影响本条决定）
- **隐私政策/用户协议入口** ✅ 需要：微信小程序上架基本强制要求展示，这不是产品自选项而是平台合规底线，原文档此前完全未提及
- **产品理念展示 / 反馈入口 / 基础设置（语言、字体、无障碍）** ❌ 均不做
- **UI 呈现形式未定**：角标点开后是弹层列表还是独立页面，仍待设计，本轮只锁定"内容"范围，不涉及呈现方式

### 6.3 第一步：呼吸引导（v6 更新：新增产品语境说明）
- 动画/动效引导深呼吸或五感接地练习
- 用户状态不好自然会跟随；状态不错自然会跳过
- 设置可跳过按钮
- **v6 新增**：呼吸圈上方常驻一句产品语境说明，永久显示（非仅新用户）：
  > "这里每天有一些小事，去做做看，感受一下就够了。"
  - 解决"为什么突然一上来就问想在哪做"的用户困惑——给用户提供"我在哪里、接下来会发生什么"的锚点
  - 同时隐含"不在乎完成"的产品立场（"感受一下就够了"暗示感受才是目的，不需要显式说"不用完成"）
  - 文案方向：只说正面的，不通过否定"完成/打卡"来强调立场（否则反而放大了这件事）

### 6.4 第二步：地球 online 卡片（v6 更新：从"每次选场景+推一条"调整为"每日卡片+候选池"）

**场景标签（scene_tags）**：在首次使用时的基本信息设置中一次性确定，作为用户档案的固定属性，不在每次打开时重新选择。用户的生活动线相对固定（上班的人大多数时候都在工位/通勤/食堂），每次都选反而制造不必要的决策负担；若场景变化，可在设置里更新。

**11 维 scene_tags 预定义集合**（来自 `content_principles.md §五`）：

| 标签值 | 对应场景 |
|---|---|
| `workspace` | 工位 |
| `classroom` | 教室 |
| `home` | 自己的房间 |
| `transit` | 地铁 / 公交 |
| `walking` | 步行 / 骑行 |
| `driving` | 私家车 |
| `convenience-store` | 便利店 |
| `canteen` | 食堂 |
| `gym` | 健身房 |
| `market` | 菜市场 |
| `general` | 通用（无特定场景要求，候选池不足时补充） |

**推送逻辑**：从推送层内容池（每日任务内容池）中，按用户 scene_tags 过滤 + 去重（排除已领取/已完成）随机抽取若干候选，**与图鉴解锁状态完全无关**；以地球 online 卡片形式展示，用户从候选中选一个领取；可"换一批"刷新（最多3次，见 5.1.1）

**推送层内容库**：当前 42 条（日常任务），覆盖上述 11 个 scene_tag 维度，每维度 ≥ 5 条，均通过 Gate 0 检验；**现有条目 scene_tags 字段需按新分类重新映射**（原 室内短/室内久/室外 已废弃）

**配套文案语气**：做歪了也算数，过程比结果重要

### 6.5 第三步：完成判定 + 对话与归档机制（v3 全面重设计）

#### 6.5.0 "完成"的判定标准

**显式自报**：用户去做完之后回到 App，点击一个低摩擦按钮（文案需用包容语气，如"做完啦"，不用"标记完成"等系统化措辞）。这个点击本身就是完成信号，**不验证、不评价**，哪怕只做了几秒也可以点（"做歪了也算数"同样适用于这个按钮）。

**为什么不能用"必须聊天"作为完成标准**：聊天是否发生，应该完全独立于"用户是否真的去尝试了这件事"这个判定。把两者绑定，会让"聊聊"从一件自愿的事变成隐性任务，违反不强迫、不评价的原则。

**为什么不能用纯隐式判断（如"打开过内容页面即算完成"）**：完成度数值依赖"完成"这个信号的真实度，如果"打开=完成"，用户随手划走也会被计为完成，完成度就失去了"投入够不够深"这个最基本的判别意义。

#### 6.5.1 聊聊的发起方式

用户点击"做完啦"后，弹出邀请——**前端展示一段统一的固定文案**（如"刚才做的这件事给你带来了什么感受吗？很愿意听你聊聊"），不依赖 AI 实时生成。用户可以选择进入对话，也可以直接跳过。

- 全产品统一一句，不分内容类型（MVP 阶段简化决定；图鉴内容数据结构中原计划的 `starter_cue` 字段保留作为未来增强点，暂不使用）
- **聊聊功能对推送层和图鉴层一视同仁**：推送内容完成后同样可以聊聊，但这类对话**不挂靠任何图鉴ID、不参与任何回顾、不生成完成摘要**，退出后数据安静保留但不处理——推送层的聊聊只服务"单次的陪伴"，不是所有感知都需要被收集进某个集合才有意义

#### 6.5.2 对话隔离与摩擦点记忆（v3 更新）

- 每个丰容内容（每次完成事件）拥有**独立对话上下文**，不与其他内容/其他次共享
- 重做同一内容（图鉴重做，或推送层池子重置后再次抽到同一条）= 全新独立对话窗口，但系统会自动传入上次同一内容的"完成摘要"（见下），让 AI 有连续性意识
- **完成摘要生成机制（仅图鉴层对话）**：用户点击「说完了」退出时，自动静默触发一次**额外的轻量 LLM 调用**（与主对话模型分离，可用更便宜的模型，如 DeepSeek，专门处理这类简单总结任务），生成一段**100字以内**的摘要，关联 (内容id + 完成时间) 存储
- 为什么不用无LLM的原文截取：用户原话往往碎片化，直接截取很难反映"这次体验的核心"
- 推送层对话不触发此机制：推送层聊聊没有下游消费者（不参与图鉴回顾），退出后直接关闭
- 这份摘要同时服务两个用途：① 重做/再推送同一内容时喂给AI做连续性参考 ② 图鉴回顾叙事生成的输入素材（多次完成的摘要能体现变化轨迹）

#### 6.5.3 对话结束机制（v8 重新设计：归档对用户不可见，改为"说完了"退出）

**设计变更原因**：归档是数据管道机制，被暴露成了用户需要理解的UI功能。原有的"归档"按钮、5条消息弹窗，用户不知道为什么要操作，而归档的下游价值（生成摘要供图鉴回顾使用）对用户完全不可见。从第一性原理看，用户结束对话的自然时刻是"决定离开、继续去做别的"，这个时刻应该成为唯一的触发点。

**新机制**：

```
对话进行中
     │
对话内常驻"说完了"按钮（随时可点，替代原"归档"按钮和5条消息弹窗）
     │
     ▼
用户点击"说完了"
     │
     ├── 展示一行固定收尾文案（如"这次说的，都在了"），短暂后淡出
     │
     └── 后台静默执行：
         图鉴层对话 → 生成摘要（DeepSeek，≤100字）→ 归档标记
         推送层对话 → 直接关闭，不生成摘要，不写归档标记
```

- **收尾文案**：固定文案，不由AI生成，措辞遵守"不评价表现"原则（不能有"你今天做得很好"类句子）。参考：「这次说的，都在了」
- **推送层对话不归档**：推送层的聊聊服务于单次陪伴，没有下游消费者，退出后数据保留但不处理
- **图鉴层对话**：退出时静默生成完成摘要，关联(内容id+完成时间)存储，作为图鉴回顾的素材
- **回顾触发后的被动归档**：图鉴100%触发回顾（快照定格）后，当时仍未退出的图鉴层对话统一被动归档+生成摘要；这些摘要不进入本次回顾（已快照），但保留备用
- **用户意外关闭小程序（未按"说完了"）**：对话保持未归档状态，不参与图鉴回顾素材。在图鉴100%时被被动归档兜底。不做onHide自动归档，避免用户临时切出去又回来时意外丢失继续聊的机会
- **决策过程（避免重复纠结）**：v2阶段否决了AI检测结束信号；v3改为按钮+计数规则；v8进一步简化：去掉5条消息弹窗，去掉"归档"这个对用户可见的概念，改用语义更直接的"说完了"作为唯一结束入口


**支持输入方式**：
- 文字输入（直接传给模型 API）
- 图片输入（**完全可选**，UI 上在文字输入框旁放置上传图标，类似聊天应用常见的"+"/相机图标，点击才弹出选图，不点击不影响文字消息发送）
- 语音：提醒用户使用手机自带语音转文字功能，再粘贴文字输入（MVP阶段方案）

**AI回应人格设定**：
- 角色：温柔的见证者，不是教练，不是治疗师
- 可以对用户分享的内容表达真实的惊喜和好奇
- 不评价用户"做得好不好"或"完成了没有"——不说"很棒"、"你做到了"
- 区分：对内容/发现的回应（✅）vs 对用户表现的评价（❌）
- 不引导"下次可以……"，只在乎这一次
- 如果用户分享图片，先说说在图片里看到了什么
- 回应控制在3句话以内，留空间给用户
- 语气像一个安静的朋友，不像一个app
- **v3 更新**：AI 不再需要承担"检测结束信号"的判断职责（该机制已改为按钮+计数规则，见 6.5.3），system prompt 可以简化，不再要求模型输出任何隐藏标记
- **System Prompt 待办（仍未执行，留待 system prompt 整体设计阶段）**：处理"对话被搁置很久后用户突然回来续聊"的语气问题——原默认"用户刚刚完成"的语境在长间隔后不成立

**对话存档用途**：图鉴层对话在退出（「说完了」）或图鉴100%被动归档时生成完成摘要，服务于连续性记忆和图鉴回顾叙事生成。推送层对话退出后不生成摘要，仅服务单次陪伴。

## 八、技术实现路径（v3 重新确认）

### 8.1 身份模型与数据存储阶段划分（v6 更新：平台已迁移至微信小程序）

- **MVP 阶段（当前）**：uni-app (Vue3 + Vite) 微信小程序，目标平台 mp-weixin，使用微信开发者工具导入 `dist/build/mp-weixin` 构建产物预览。处于内测阶段，不做公开部署。所有状态用微信 `wx.setStorageSync / wx.getStorageSync` 本地持久化（等效于原 localStorage 方案，语义完全对应），跨启动保留——可测试任务池去重、已完成追踪、历史记录等依赖"跨时间状态"的核心体验。
  - **原描述（v4）的"H5/localhost + localStorage"已被微信小程序 + wx.storage 替代**，架构决策不变
  - 不接 Supabase，不做匿名设备 ID / 真实账号方案（MVP 阶段不需要）
  - 已接入真实 API：和风天气（天气/城市）、Qwen（主对话）
- **生产阶段**：部署至微信小程序正式版，用户登录微信账号（openid）后同步与持久化数据（图鉴解锁状态、完成度、解锁名额余额、对话记录及归档状态）。届时再评估是否需要 Supabase 或微信云开发自带的数据库

### 8.1.1 主动提醒的实现约束（v4 新增，呼应 6.2.1）

生产阶段计划用微信小程序「订阅消息」API（`wx.requestSubscribeMessage`）实现，**不需要绑定公众号/服务号**（订阅消息是小程序独立能力，与公众号「模板消息/服务通知」是两套不互相依赖的体系）。但有几条硬限制需要提前写进 PRD，避免开发阶段才发现做不到：

- 不能后台随时主动推送：必须先有用户一次主动触发（如完成内容后的"做完啦"流程）弹出授权询问，用户同意后产品方才能补发一条
- 默认是一次性订阅消息，每次发送后授权即消耗，需要再次提醒得再走一次授权流程；长期订阅消息（一次授权可多次发送）仅开放给政务/医疗/交通/金融/教育类公共服务类目，本产品不适用
- 消息文案受微信模板库审核限制，不能完全自由发挥产品语气，只能在审核通过的模板字段范围内选最贴近的措辞
- 用户接收提醒的开关在微信「服务通知」会话列表里由用户自行管理，小程序内做不出一个可控的"提醒开关"，只能做"引导用户去授权"的入口

### 8.2 后端架构

- **MVP 阶段**：不搭后端代理，前端直接调用模型 API（本地测试，无公开部署，API Key 暴露风险面极小）
- **生产阶段（小程序）**：小程序无法直接调用第三方 API，需自建后端代理（或用微信云函数），且需域名白名单等配置。**这套代理 MVP 阶段不提前搭建**——其设计强烈依赖小程序云开发的具体形态，提前搭建很可能在迁移时被重写，等于白做

### 8.3 模型选型（v3 确认，替代 v2 的 Anthropic Claude 方案）

- **主对话模型（文字+图片理解）**：统一使用具备视觉理解能力的 Qwen 系列模型（如 qwen3.7plus），**不做模型路由**——同一个模型处理纯文字和含图片的对话，避免"文字用模型A、图片用模型B"导致语气/人格表现不一致的问题
- **摘要生成模型**：使用更轻量、成本更低的 API（如 DeepSeek）专门处理"对话归档时生成100字摘要"这类简单总结任务，与主对话模型解耦。理由：这是用户完全无感知的后台调用，不需要承担"人格一致性"的负担，适合用便宜模型

### 8.4 文字输入
用户输入文字 → 拼接 system prompt + 用户文字 → 调用 Qwen API → 流式返回回复

### 8.5 图片输入（已确认保留，UI 见 6.5.3）
用户上传图片 → 前端转base64（或对应 Qwen API 要求的格式）→ 作为图片内容块传入对话 → 模型结合图片+文字回应
**注意**：图片建议压缩到1MB以内，支持jpg/png/gif/webp
**亮点**：用户拍一张color walk的照片，AI能真的"看见"它并回应，体验惊喜感强

### 8.6 语音输入（MVP方案，不变）
提醒用户使用手机自带的语音转文字功能，再将文字粘贴输入。后续版本可视情况接入语音转文字API。

## 十、设计原则备忘

- **不制造焦虑**：任何功能（推送频率、打卡、streak、进度条）都要检查：它有没有在暗示"你做得不够好"
- **感知先于记录**：所有涉及"分享"或"记录"的设计，顺序是感知在前，记录在后
- **丰容不是习惯养成**：不强调坚持，不强调完成率，只在乎有没有真实体验发生
- **温婉包容**：产品语气始终包容不完美，做歪了也算数
- **推送而非选择**：用户的认知边界限制了他能想到什么，产品的价值在于帮他看见他本来看不见的可能性
- **数值不可见原则**：凡是涉及完成度、进度、计数的内部判定逻辑，一律不向用户暴露具体数字/百分比/进度条，只用语气包装结果（如"还没到时候""差不多了"）
- **（v3 新增）功能服务于什么优先于功能怎么做**：讨论任何机制前，先确认它对结果有没有硬性影响（改变了什么数据/状态、不存在会怎样）。如果一个机制没有硬性影响（如归档、回顾叙事的触发精度），就不必追求判断/实现的高精度，把工程量花在"是否传达了正确的意图/体验"上。这条原则已同步写入用户的全局 CLAUDE.md，作为所有产品讨论的默认思考范式

## 十一、System Prompt 设计定稿（v5 新增）

三段 system prompt 涉及的结构性决策（注入变量、触发时机、语气边界）均已通过 grill-me 梳理确认，文本本身也已写出第一版并经过模拟测试验证。

### 11.1 主对话人格（触发于"做完啦"后用户选择"聊聊"）

**结构性决策**：
- 注入变量：内容标题+instructions（必须）、历史完成摘要（仅redo场景，有则带，没有则省略该字段）；**不**注入场景/图鉴元数据（讨论不出它对对话质量有什么实质影响，更像是给系统检索/去重用的，不是给人格调语气用的）
- 开场方式：具体回指刚才内容的细节 + 开放邀请，不强制要求回答（不用"直接抛一个具体问题"那种访谈感更重的方式）
- 行为边界：不评价表现/不给建议，只反映和追问用户说的内容；"不下结论"也包括不要给用户的感受加自己的解读定性（这本质上是说教的隐藏形式）；单纯反映不带问句本身就合格，不必每次都追问
- 长间隔后回来续聊：不做特殊语气处理，复用现有对话历史即可——结束对话靠用户主动点「说完了」，不是时间自动判定，所以「隔很久」本质是同一对话被打断后续上，内容已经在那一刻被捕捉，时间跨度不改变该聊什么
- 职责边界：AI 的语气不承担「暗示该结束了」的功能，那是「说完了」按钮的工作；AI 只管诚实反映对话内容本身的浓淡，不因轮次增多主动把语气往收尾方向带

**定稿文本（v7 改写：正面人格版）**：
```
你是用户聊天的那个安静陪伴者——不是教练，不是治疗师，
就是一个刚好在场、真的在听的朋友。

用户刚完成的事：
标题：{{content_title}}
内容：{{instructions}}
{{#if previous_summary}}
ta上次做这件事留下的印象：{{previous_summary}}
{{/if}}

开场时，具体回指这件事里某个细节，发出一个开放的邀请——
不一定是问句，留空间让ta自己说。

之后，把注意力完全放在ta刚说的那句话上。
呼应它，追问它，或者只是安静地把它接住。
不引向别处，不给下一步建议，不帮ta总结
"这件事对你意味着什么"——你只在乎ta说的这一句。

如果ta分享了图片，先说说你在里面看到了什么。
{{#if previous_summary}}
如果有自然的连接点，可以轻轻提一下上次的印象，不要数次数。
{{/if}}
回应控制在3句话以内。

一条绝对的边界：不对ta的表现做任何判断——
不说"你完成了""你坚持下来了""你做得很好"，
哪怕是表扬也不行。
```

### 11.2 摘要生成（触发于对话归档时，轻量模型）

**结构性决策**：
- 输入：完整对话记录+标题/instructions
- 无对话记录时（用户只点"做完啦"没"聊聊"）：留空，不强行生成占位摘要——完成事件本身由"做完啦"按钮独立记录，跟摘要表分开，回顾时仍知道"做了这件事"，只是没有对应的对话细节可引用
- 风格：避免评价/结论性语言，只保留用户说的具体内容

**定稿文本**：
```
你的任务是把下面这段对话总结成一段不超过100字的摘要。

这次完成的内容：
标题：{{content_title}}
内容：{{instructions}}

完整对话：
{{conversation}}

要求：
- 只保留用户自己说过的具体内容（看到的、感觉到的、说的话），不要加你自己的评价或结论
- 不用"开心""有所收获""感到平静"这类概括性的情绪定性词，用用户原话里更具体的描述（比如"注意到叶子背面是另一种绿"，而不是"用户感受到了自然之美"）
- 不超过100字，不用完整段落的修饰语，直接、简洁
- 如果对话内容很少或很碎片化，摘要也可以很短，不需要硬凑满

直接输出摘要文本，不要加任何前缀说明。
```

### 11.3 回顾叙事生成（触发于图鉴完成度100%）

**结构性决策**：
- 输入范围：上一次回顾以来的所有新完成摘要（第一次回顾则是图鉴下全部摘要）
- 两条并存的连续性机制：
  - **机制A（内容层面，§5.4.1）**：每次生成回顾都适用——针对本批摘要里出现的、过去也做过的 content_id，取该内容自己的历史摘要序列，识别"反复做同一件事"的模式
- 三层取数逻辑：① 多次完成且都有摘要→取全部历史摘要做模式识别 ② 仅完成一次且有摘要→直接用这一条 ③ 完成过但没摘要→只用标题/instructions代入，告诉回顾"做了这件事"，没有感受细节可引用
- **关键写作要求（v5 新增，经模拟验证）**：不要按完成顺序逐条复述发生了什么（会变成事件流水账），写作前先从摘要里找一条贯穿的线索（用户的感受/注意力呈现出的某种倾向），用这条线索组织整段叙事，具体细节只作为支撑这条线索的例证出现
- 不做结构化情绪统计/聚类（§5.4.1既有约束）；语气边界遵循§5.4.2（可呼应不可评价表现，立场不可模板化）；摘要全空时使用§5.4.2给定的固定兜底模板

**定稿文本**：
```
你的任务是为用户写一段完整的回顾叙事，纪念ta在"{{collection_name}}"这本图鉴里完成的探索。

这是一段连续的、有温度的叙事文字，不是总结报告，不是数据列表。你的语气目标是"一起回望"——像被陪伴、被看见的感觉，不是写总结陈词。

{{#if has_summaries}}
这是ta这段时间在这本图鉴里留下的记录（按完成的内容分组，同一条内容如果做过多次，会附带每一次的摘要）：

{{items_data}}
（每条内容的格式：
- 《内容标题》

写作前，先从这些摘要里找一条贯穿的线索——ta的感受或注意力呈现出的某种倾向、生活里被重新看见的某个角落，不是"完成了哪些事"本身。用这条线索组织整段叙事，具体细节只作为支撑这条线索的例证出现，不要按完成顺序逐条复述发生了什么，那会变成事件流水账，不是叙事。

如果同一条内容下出现了多条摘要，这是需要你识别和呼应的一种模式——不要机械列出或一一对比，而是像一个真正记得这些细节的人那样，自然地在叙事里点出"你好像对这件事有种特殊的偏好"之类的呼应。不要做任何形式的情绪分类、打标签或频次统计（比如不要写"3次平静、2次惊喜"），所有的模式识别都必须是定性的、融合进叙事语言本身的。
```
