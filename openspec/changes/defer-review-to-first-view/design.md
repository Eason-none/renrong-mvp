# Design: 回顾快照推迟到首次查看时生成

## Context

原时序（做完啦瞬间：先快照→后被动归档→棘轮）为的是"素材可复现 + 不打断轻量流程"。可复现性真正的承载者是棘轮（生成一次永不重来），而非"在哪个瞬间生成"；不打断流程在惰性生成下反而更彻底（完成瞬间零后台调用）。

## Goals / Non-Goals

**Goals:** 最后一条的聊聊能进入回顾；失败可自愈（下次点开重试）；完成瞬间零 API 调用。
**Non-Goals:** 不改回顾文本的生成 prompt/兜底模板；不改棘轮单向性；不做进行中的进度提示。

## Decisions

1. **新时序（`ensureFirstReviewSnapshot`，首次点开回顾时调用）**：入口同步定格该图鉴未归档对话名单 → 逐个被动归档（其摘要纳入素材）→ 收集全部摘要 → 生成快照落盘 → 置棘轮。任一步失败即整体抛出：不落快照、不置棘轮，已完成的单条归档保留（幂等，重试时跳过）。
2. **重试即"再点开一次"**：ReviewView 每次打开时若无快照就调用 ensure；原"App 启动补偿"契约删除（从未实现，现在也不需要了）。
3. **点开时新开对话的竞态**：名单入口定格（沿用上一个 bug 的修法）——生成期间用户若返回图鉴另开对话，该对话不被归档、不进本次素材，留待其自然归档；快照已定格不再追加，可接受。
4. **卡片入口**：completed 状态恒显"✦ 已点亮 回顾 →"。加载文案沿用邀请式定稿文本；生成失败时换温婉重试文案（不带技术字样）。
5. **explore 页"回顾"总入口计数**：维持按已生成快照数统计——未点开过的图鉴回顾尚不存在，不计入，点开后自然纳入。

## Risks / Trade-offs

- [首次点开需等待生成（数秒）] → 加载文案本就是为此设计的（§5.4.2 邀请式语气）
- [摘要与叙事跨多次 API 调用，中途失败重试会重复已成功的归档] → archiveConversation 幂等 no-op，无重复摘要
- [历史数据：已按旧时序定格的快照] → 不迁移不重生成，棘轮语义如常

## Migration Plan

无数据迁移。已定格快照维持原样；卡在"有快照但棘轮未置位"旧中间态的图鉴，ensure 的既有兜底（发现已有 sequence=1 快照则跳过生成、补做归档+棘轮）会在下次点开时自动收敛。
