# 指标体系（add-metrics-system）

## Why

产品即将上线，但所有行为数据仅存于用户本地 storage，没有任何服务端采集——上线后既看不到"有没有人用"（运营基本盘），也无法验证产品核心假设（"普通时刻因产品变真实了一点"），而内容扩产的解冻条件正是用户反馈。指标体系就是那个被等待的反馈通道，必须在上线前就位，否则最关键的早期种子用户行为将永久丢失。

## What Changes

- 新增客户端匿名事件上报：最小事件集 3 个——`session_start`（打开）、`task_completed`（完成，带类型 + 内容 ID）、`review_opened`（回顾被打开）。不采任何中间步骤和页面级事件。
- 新增本地随机匿名 ID（UUID，首次启动生成后持久化），作为唯一上报标识。用户手填的 `player_id`（称呼）**不可**用于上报。
- Supabase 新增最小 `events` 表接收上报；运营基本盘（UV/留存/页面）全部交给微信小程序数据助手，零自建。
- 隐私边界：上报载荷仅含 匿名 ID + 事件名 + 事件属性（类型/内容 ID）+ 时间戳。称呼、生日、场景标签、对话文本、回顾快照永不出设备。小程序隐私保护指引需声明"使用记录"。
- 新增仓库文档 `metrics.md`：北极星与诊断切片的 SQL 写死在文档里，每周手动跑；不建看板、不做自动报告。
  - 北极星：首次使用后 7 天内自发回访 ≥1 次且再次完成任务。
  - `review_opened` 口径：点亮过图鉴人群中的回顾打开率（分母由 `task_completed` + 静态内容库推导）。
  - 反指标清单写死：streak/连续天数、人均完成量目标、会话时长最大化、完成率 KPI 永不计算、永不进报表。

## Capabilities

### New Capabilities

- `analytics-events`: 客户端匿名行为事件的采集与上报——匿名 ID 生成、三个事件的触发点与载荷、隐私边界（永不上传的数据清单）、上报失败的静默处理（绝不阻塞或打扰用户流程）、反指标约束。

### Modified Capabilities

（无——事件采集挂接在现有流程的完成点上，不改变任何现有 capability 的行为要求。）

## Impact

- **代码**：新增上报模块（匿名 ID + 事件发送）；在应用启动、任务完成、回顾打开三个既有代码路径上各加一次调用。
- **Supabase**：新增 `events` 表（含 RLS：客户端只写不读）。
- **配置**：微信小程序后台隐私保护指引新增"使用记录"声明。
- **文档**：新增 `metrics.md`（指标定义 + SQL + 反指标清单）。
- **不受影响**：现有本地数据结构、所有用户可见界面与交互（上报完全静默）。
