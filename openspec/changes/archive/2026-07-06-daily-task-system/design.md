## Context

项目为 uni-app (Vue3 + Vite) 微信小程序（目标平台 mp-weixin）。当前已有两条内容轨道：推送层（场景→随机推送卡片）和图鉴层（图鉴条目解锁/完成）。每日任务是完全独立的第三条轨道，与前两者无状态共享、无完成度联动。

外部 API 层现有 Qwen（对话）和 DeepSeek（摘要），本次新增 GPS 城市定位（wx.getLocation）和天气查询接口（和风天气或类似，提供城市级当日天气状况描述）。

## Goals / Non-Goals

**Goals:**
- 每天首次打开 App 时弹出日推卡片（地球Online风格），展示玩家信息、城市天气、固定文案、当日三个候选任务
- 用户可从候选任务中主动领取，领取后进入个人任务池，可随时完成，不过期
- 基本信息设置（玩家 ID、出生日期、场景偏好）一次录入，同时服务日推卡片个性化和任务内容匹配
- 每日任务内容池独立于推送层和图鉴层，按场景标签匹配用户偏好

**Non-Goals:**
- 每日任务与推送/图鉴完成度的任何联动
- 服务端调度或推送通知（依赖用户主动打开 App）
- AI 动态生成每日任务内容
- 任务完成统计、连续打卡、排行榜等游戏化机制
- 用户自填场景标签的 NLP 识别

## Decisions

### D1：日推卡片出现时机用本地日期比对，不依赖服务端

存储 `DailyCardShownDate`（格式 `YYYY-MM-DD`）。App 启动时与当日日期比对，不同则展示卡片并更新该 key。

**Alternative considered**: 用时间戳 + 24h 窗口。拒绝原因：跨日的直觉是"今天/明天"，不是"24小时后"，凌晨1点仍算新的一天。

### D2：当日三个候选任务在卡片展示时从内容池里随机抽取，不持久化"今日候选"

候选任务仅在卡片展示期间存在（内存状态），关闭卡片后不保存未领取的候选。已领取的任务才写入 `DailyTaskPool`。

**Alternative considered**: 持久化今日候选，用户关闭后重开 App 还能看到同一批。拒绝原因：增加状态复杂度，用户已领取的永不消失已能解决"找不到"问题，未领取的消失不产生损失感。

### D3：已领取任务在「当下」tab 中以独立分区呈现，不新增 tab

`index.vue`（当下）在现有 BreathingGuide/场景/推送流程之外，新增「我的日常任务」折叠区块，展示所有已领取但未完成的任务。点击任意一条直接进入做完啦→聊聊流程。

**Alternative considered**: 新增第三个 tab 或 NavBar badge。拒绝原因：日常任务量少（每次领取1-3条），独立 tab 过重；NavBar badge 的 📖⚙ 已经较密，再加一个拥挤。折叠区块在「当下」tab 里语义自洽（每日的事都在"当下"里找）。

### D4：场景匹配用标签交集，任意命中即纳入候选池

用户三个维度的所有勾选项形成一个 scene_tags 集合，内容条目的 scene_tags 与之有任意交集则进入候选池，从候选池随机抽 3 条。若候选池不足 3 条，用通用标签（`general`）的条目补足。

### D5：天气 API key 存 VITE 环境变量，定位拒绝时优雅降级

城市显示"未知"，天气显示"未知"，卡片其余内容正常展示。不因权限缺失阻断卡片出现。

### D6：BasicInfo 未录入时日推卡片仍可展示，使用占位值

玩家 ID 未填 → 显示"旅行者"；出生日期未填 → 不显示生存天数行；场景偏好未填 → 从通用池抽取任务。卡片末尾加引导"去完善你的信息"跳转基本信息设置。

## Risks / Trade-offs

- **wx.getLocation 首次需弹出授权弹窗** → 卡片加载时异步请求，授权失败降级为"未知"，不阻塞卡片渲染
- **天气 API 调用失败**（网络/额度）→ 同上，降级显示"未知"；VITE key 未配置时直接跳过请求
- **已领取任务池无限增长** → MVP 阶段不设上限；内容库条目数量本身有限，实际不会有问题
- **「当下」tab 已有较复杂的状态机（BreathingGuide→场景→推送）**，折叠区块的插入位置需谨慎，避免打断主流程视觉节奏

## Open Questions

- 天气接口选型（和风天气免费版、心知天气等）需在实现前确认，key 申请是用户侧操作
- 每日任务内容库初始条目数量——建议每个场景标签至少 5 条，总计 30-50 条作为 MVP 内容基准，但内容撰写是产品侧工作，不在本次实现范围内（实现阶段先用占位内容）
